# Self Evaluation – Slime Laboratory Management System

## Student Reflection

This project was developed to demonstrate Object-Oriented Programming principles through the design and implementation of a Slime Laboratory Management System.

The system allows different slime types to be created, managed, enhanced through environmental effects, and combined within a laboratory environment. The project was designed using object-oriented principles to ensure the code is reusable, maintainable, and extensible.

---

## Criterion 1 – Object-Oriented Design

The project demonstrates several Object-Oriented Programming concepts.

### Abstraction

The `Slime` class is implemented as an abstract base class. It defines common behaviour and attributes shared by all slime types while enforcing implementation of slime-specific functionality through abstract methods.

Abstract methods:

* `get_slime_type()`
* `get_special_attributes()`

This ensures that every slime subclass provides its own implementation.

### Inheritance

Inheritance is demonstrated through:

* `ShadowSlime`
* `StormSlime`

Both classes inherit from the `Slime` class and reuse common functionality while extending the behaviour with their own unique attributes and methods.

### Encapsulation

Encapsulation is implemented using property methods and validation.

Examples include:

* Validation of slime IDs
* Validation of slime size values
* Controlled access to object attributes

This prevents invalid data from being assigned to objects.

### Polymorphism

Polymorphism is demonstrated through the use of abstract methods that are implemented differently by each slime subclass.

The `Laboratory` class can interact with different slime types through the common `Slime` interface without needing to know the exact subclass being used.

### Composition

Composition is demonstrated through the relationship between the `Laboratory` and `Slime` classes.

The laboratory stores and manages multiple slime objects throughout the application.
